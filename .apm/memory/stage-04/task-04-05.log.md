# Task 4.5 - Persistir nome do responsável e corrigir o pedigree

## Summary
Investigado e corrigido o achado da Task 4.3: o "Nome completo" coletado no formulário de cadastro nunca era enviado ao Supabase Auth. Agora é persistido em `user_metadata.full_name` e usado no certificado de pedigree, com o e-mail como alternativa apenas para contas criadas antes desta correção.

## Details
1. Investigação confirmou a causa raiz: `app/login/page.tsx` coleta o campo `name` ("Nome completo") no formulário de cadastro (linha 127), mas `signUpAction` em `lib/auth/actions.ts` só lia `email`/`password` via `readCredentials()` — o nome era descartado silenciosamente, nunca chegando a `supabase.auth.signUp`.
2. Optei por Supabase Auth `user_metadata` em vez de um campo novo no `Tenant`, conforme a preferência sugerida pela Task — é o local natural para um dado de perfil do usuário autenticado (distinto do `Tenant`, que representa o criatório/conta), e evita uma migração de schema para um dado que o próprio Supabase Auth já suporta nativamente.
3. `lib/auth/signup.ts` (novo arquivo): `buildSignUpPayload(formData)` monta `{ email, password, options: { data: { full_name: name } } }`. Extraído para um módulo separado (sem `"use server"`) porque arquivos com essa diretiva exigem que toda exportação seja uma Server Action assíncrona — uma função pura auxiliar não é permitida no mesmo arquivo que `signUpAction`.
4. `lib/auth/actions.ts`: `signUpAction` agora usa `buildSignUpPayload(formData)` para construir a chamada a `supabase.auth.signUp`.
5. `lib/pedigree/responsavel.ts`: `resolverNomeResponsavel` passou a preferir `user.user_metadata.full_name` (com `trim()`, tratando strings vazias/em branco como ausentes), caindo para o e-mail quando ausente, e para "Responsável não identificado" quando nenhum dos dois existir — preserva o comportamento de alternativa da Task 4.3 para contas já existentes.
6. Testes automatizados:
   - `lib/auth/signup.test.ts`: testes unitários de `buildSignUpPayload` cobrindo a persistência do nome em `options.data.full_name`, remoção de espaços em branco, e o caso do campo nome ausente.
   - `lib/pedigree/responsavel.test.ts`: testes unitários de `resolverNomeResponsavel` cobrindo os quatro casos — nome persistido presente, nome ausente (alternativa por e-mail), nome em branco (alternativa por e-mail), e nenhum dos dois presente (rótulo genérico).
   - Não foi adicionado um teste de integração que efetivamente crie um usuário real no Supabase Auth: sem uma chave de service role disponível neste projeto, não há como limpar esses usuários de teste depois, o que acumularia contas órfãs no projeto Supabase compartilhado a cada execução da suíte. A cobertura unitária de `buildSignUpPayload` já garante que o payload correto é enviado; o comportamento de que `options.data` é persistido como `user_metadata` é um contrato documentado do próprio Supabase Auth, não uma lógica escrita neste projeto.
