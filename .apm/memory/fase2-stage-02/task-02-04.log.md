# Task 2.4 - Tela de revisão/edição da importação

## Summary
Fluxo completo de importação do IBAMA implementado: upload do PDF → revisão/edição de cada linha extraída → confirmação explícita → gravação no plantel. Nenhuma escrita acontece antes da confirmação final do usuário.

## Details
1. Validado o estado real de `lib/importacao-ibama/`, `lib/aves/compatibility.ts` e das telas relevantes antes de implementar — confirmado que nenhuma UI de importação existia ainda (Tasks 2.2/2.3 eram só serviços de backend); esta Task precisou construir também o ponto de entrada (upload) além da revisão em si, já que nada mais no pipeline os invoca.
2. **Decisão de posicionamento** (não especificada explicitamente na Task): a tela vive em `/configuracoes/importar-ibama`, com uma entrada em Configurações ("Importar do IBAMA", nova seção "Plantel") — escolha alinhada com a opção "ação separada acessível a qualquer momento, em Configurações" citada em `docs/solicitacao-fase2.md` como uma das alternativas em aberto. Documentado aqui por ser uma decisão de UX não coberta literalmente pelo Task Prompt.
3. `lib/aves/require-tenant.ts` (novo): extraí `requireTenantOrRedirect` (antes uma função local de `lib/aves/actions.ts`) para um módulo compartilhado sem a diretiva `"use server"` — necessário porque um módulo `"use server"` só pode exportar funções assíncronas invocáveis diretamente pelo cliente, e a nova Server Action de importação (`lib/importacao-ibama/actions.ts`) precisava reaproveitar exatamente essa mesma guarda de autenticação/tenant.
4. `lib/aves/schema.ts`: `createAveSchema` (e por extensão `updateAveSchema`) ganhou `nomeCientifico`, `tipoAnilha`, `diametroAnilha` como campos opcionais — esses campos existem no schema Prisma desde a Task 2.1, mas nunca tinham sido aceitos por nenhum fluxo de criação/atualização até esta Task, que é quem efetivamente os persiste pela primeira vez.
5. `lib/aves/service.ts`: `buscarAvePorAnilha(anilha)` (novo, tenant-scoped automaticamente pelo middleware de isolamento) — usado para detectar duplicata antes de qualquer gravação.
6. `lib/importacao-ibama/actions.ts` (duas Server Actions):
   - `processarPdfIbamaAction`: recebe o PDF via `FormData`, chama `extrairDadosIbama` (Task 2.2) → `resolverEspeciesDasAves` (Task 2.3) → para cada ave, verifica duplicata de anilha no tenant atual via `buscarAvePorAnilha`. Retorna as linhas com um `linhaId` gerado no servidor (para chaves React/edição no cliente), `origem` sempre pré-selecionada como "Adquirida" (regra desta fase — documento do IBAMA não informa origem), e a flag `duplicada`/`aveExistenteId` quando aplicável.
   - `confirmarImportacaoAction`: recebe o array de linhas (já editadas pelo usuário) e uma confirmação opcional de responsável; delega a gravação a `executarImportacao` (ver item 7) dentro de `runWithTenant`; só grava a sugestão de nome (`user_metadata.full_name`) ou telefone (`Tenant.telefone`) quando o payload explicitamente os inclui — nunca automaticamente.
7. `lib/importacao-ibama/importar.ts` (novo): `executarImportacao(linhas)` extraída da Server Action para ser testável isoladamente sem depender do contexto de cookies/sessão do Next.js — mesma técnica de extração já usada para `buildSignUpPayload` (Task 4.5). Para cada linha: se `duplicada` e `atualizarExistente` confirmado pelo usuário, chama `updateAve` no registro existente; se `duplicada` sem confirmação, **pula a linha** (nunca cria duplicata nem sobrescreve silenciosamente); caso contrário, `createAve`. Reaproveita `createAve`/`updateAve` (Fase 1) integralmente — a validação de compatibilidade espécie/sexo de pai/mãe (`assertPaiCompativel`/`assertMaeCompativel`) já acontece dentro dessas funções, sem duplicação.
8. `app/configuracoes/importar-ibama/ImportarIbamaView.tsx`: tela client-side em duas etapas — upload (form simples com `useActionState`) e revisão (um cartão editável por ave, reaproveitando os mesmos controles de formulário do Cadastro Manual de Ave — `TextField`, toggle de sexo/origem, `Alert` — sem introduzir um padrão visual novo). Cada cartão usa `useParentesCandidatos(linha.especieId)` (o mesmo hook do Cadastro Manual, Fase 1) para os dropdowns de pai/mãe, que por construção só retornam aves já persistidas no banco (nunca outra linha do mesmo lote ainda não confirmado). Linhas duplicadas mostram um `Alert` de aviso com uma checkbox explícita "Atualizar" (desmarcada por padrão). A sugestão de responsável (nome/telefone) aparece com checkboxes de confirmação explícita, também desmarcadas por padrão. Nada é editado/gravado a partir do estado originalmente extraído — toda edição do usuário atualiza o estado local (`linhas`) antes da confirmação, e é esse estado editado (não o original) que é enviado a `confirmarImportacaoAction`.
9. Testes automatizados: `lib/importacao-ibama/importar.test.ts` (4 casos, integração contra o banco real) — criação de ave nova persistindo valores editados; linha duplicada ignorada sem confirmação (não cria, não atualiza); linha duplicada atualizada mediante confirmação explícita; persistência dos campos exclusivos da importação (nomeCientifico/tipoAnilha/diametroAnilha/registro). `lib/aves/service.test.ts`: 3 casos novos para `buscarAvePorAnilha` (encontra no tenant atual, retorna null quando não existe, isolamento entre tenants).
10. Não foi possível testar `processarPdfIbamaAction`/`confirmarImportacaoAction` diretamente (mesma limitação já documentada nas Tasks 4.5/5.1: Server Actions que dependem de `next/headers` `cookies()` não rodam fora de um request Next.js real) — a lógica de negócio extraída para `executarImportacao` está coberta; a orquestração de upload/tenant-guard nas Server Actions em si não tem teste dedicado, consistente com o padrão já estabelecido no projeto para esse tipo de plumbing.
11. Verificação visual via preview: tela de upload confirmada visualmente (fiel aos tokens do design system); nova seção "Plantel" com o link "Importar do IBAMA" em Configurações confirmada visualmente. **Não foi possível verificar visualmente a etapa de revisão** (cartões editáveis, alerta de duplicata) via automação de navegador — a simulação de seleção de arquivo real em um `<input type="file">` não é possível via script (mesma restrição de segurança do navegador já documentada nas Tasks 5.4/2.2/3.2). A confiança na fidelidade dessa etapa se apoia na reutilização direta e sem modificação dos mesmos componentes de formulário já usados e visualmente validados no Cadastro Manual de Ave (Fase 1).

## Compatibility Concerns
A etapa de revisão da importação (cartões editáveis por ave, alerta de anilha duplicada) não pôde ser verificada visualmente via automação de navegador — o upload de um arquivo real não pode ser simulado por script (restrição de segurança do navegador, não do produto; mesma categoria já registrada em Tasks anteriores desta sessão). Recomendo uma verificação manual humana pontual do fluxo completo (upload de um PDF real de teste → revisão → confirmação) antes da revisão externa.

## Important Findings
O ponto de entrada da tela de importação (onde ela vive na navegação) não estava especificado em nenhum Task Prompt anterior desta Fase — era uma decisão explicitamente deixada em aberto em `docs/solicitacao-fase2.md` para uma sessão de planejamento. Optei por Configurações (`/configuracoes/importar-ibama`), consistente com o padrão já estabelecido nesta Fase para ações de catálogo/conta (gestão de espécies, Task 1.2). Se essa não for a localização pretendida pelo usuário, é uma mudança de rota simples de revisar.
