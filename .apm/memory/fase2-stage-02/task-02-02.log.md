# Task 2.2 - Serviço de extração do PDF do IBAMA

## Summary
`extrairDadosIbama(buffer)` implementado, extraindo a tabela de aves por posição de coluna (via detecção geométrica de tabelas, sem OCR) e a identificação do responsável (nome/telefone) do PDF "Relação de Passeriformes" do IBAMA, com tratamento defensivo de linhas malformadas.

## Details
1. Avaliei bibliotecas de extração de PDF disponíveis e escolhi `pdf-parse` (v2.4.5, baseada em `pdfjs-dist`) — sua API `getTable()` detecta tabelas geometricamente (a partir das posições/bordas de texto do PDF), o que atende diretamente ao requisito "por posição de coluna, sem OCR" sem eu precisar implementar detecção de colunas do zero.
2. `lib/importacao-ibama/parser.ts`: `extrairDadosIbama(buffer)` chama `getText()` (para a seção de identificação do responsável, que é texto livre) e `getTable()` (para a tabela de aves) na mesma instância do parser, **sequencialmente** (não em paralelo — ver Important Findings). Cada linha da tabela é validada quanto ao número de colunas (8: `#, Nome científico, Nome comum, Sexo, Nascimento, Tipo anilha, Diam., Código de anilha`) e ao valor de sexo (M/F/I); linhas de cabeçalho são ignoradas via heurística (primeira coluna não numérica). Nome científico e nome comum são normalizados (`\s+` → espaço único) para tolerar quebras de linha dentro de uma célula quando o texto é mais longo que a largura da coluna.
3. Mapeamento de sexo implementado nesta Task (M→MACHO, F→FEMEA, I→NAO_SEXADO) conforme instruído, mesmo sendo a tela de revisão (Task 2.4) quem efetivamente aplica.
4. Extração da identificação do responsável: regex restrita a "Nome do Criador:" e "Telefone:", sem cruzar quebras de linha (para não capturar conteúdo de outras seções do texto por engano). CPF e endereço **não são capturados** — nem sequer uma tentativa de regex para esses campos foi escrita, para não correr o risco de coletar dado sensível por acidente.
5. Estrutura de retorno: `{ aves, linhasComErro, responsavel }` — linhas com número de colunas inesperado, sexo inválido, ou anilha ausente vão para `linhasComErro` (com o motivo) em vez de lançar exceção ou interromper a extração das demais linhas.
6. **Fixture de PDF sintético** (`lib/importacao-ibama/fixtures/gerar-pdf-sintetico.tsx`): gerada em memória via `@react-pdf/renderer` (já usado no projeto desde a Task 4.3), reproduzindo a mesma estrutura de colunas do documento oficial com dados fictícios — bordas completas em cada célula (necessárias para a detecção geométrica de tabela do `pdf-parse` funcionar). **Em nenhum momento `docs/relação de aves.pdf` foi lido, copiado ou referenciado** — usei apenas a descrição de colunas already presente no Task Prompt e em `docs/solicitacao-fase2.md` (que também alerta sobre os dados sensíveis do arquivo).
7. Testes automatizados (`lib/importacao-ibama/parser.test.ts`, 6 casos): extração correta de todas as colunas de múltiplas aves, ausência de CPF/endereço no retorno, tratamento defensivo de sexo inválido (linha vai para `linhasComErro`, outras linhas continuam sendo processadas), tratamento defensivo de anilha ausente, retorno de `dataNascimento: null` (sem lançar) para data em formato inesperado, e retorno de responsável nulo quando a seção de identificação não está presente.

## Important Findings
Durante o desenvolvimento, descobri um bug de compatibilidade real no `pdf-parse` v2.4.5: chamar `parser.getText()` e `parser.getTable()` **em paralelo** (`Promise.all`) na mesma instância do parser trava o worker interno do `pdfjs-dist` com `DataCloneError: Cannot transfer object of unsupported type` — reproduzido tanto em Vitest quanto em Node puro (não é um problema do ambiente de teste). A correção foi chamar as duas operações **sequencialmente** na mesma instância. Se uma Task futura (ex: Task 2.4, tela de revisão) reaproveitar este parser ou construir lógica similar com `pdf-parse`, deve evitar chamadas paralelas na mesma instância de `PDFParse`.
