# Task 1.4 - Design system no código e shell de navegação

## Summary
Biblioteca de componentes base (Button, IconButton, TextField, Badge, Card, Alert) extraída de `design/Design System.dc.html`, e shell de navegação responsivo (sidebar desktop / bottom nav mobile, breakpoint 900px) montado conforme `design/03 Dashboard.dc.html`, envolvendo o Dashboard existente.

## Details
1. Confirmados os tokens de tema já extraídos na Task 1.3 (`app/globals.css`, `app/layout.tsx`) como suficientes e corretos — não foram recriados, apenas reaproveitados pelos novos componentes.
2. Construídos os componentes em `components/ui/`: `Button` (variantes primary/secondary/tertiary/destructive, tamanhos default/small, estado disabled), `IconButton` (botões circulares/quadrados de ícone), `TextField` (label + input + estado de erro, com suporte a fonte mono para campos como anilha), `Badge` (variantes success/warning/danger/info/neutral/risk, com export `BIRD_STATUS_BADGE` mapeando os status de aves do Cadastro Geral), `Card` (container base) e `Alert` (warning/success/danger com ícone, título e descrição).
3. Construído o shell de navegação em `components/nav/`: `AppShell` (client component, usa `usePathname` para destacar o item ativo), `nav-items.ts` (itens da sidebar e da bottom nav, replicando exatamente a ordem/seleção de `design/03 Dashboard.dc.html`), `icons.tsx` (ícones SVG replicados do design). O breakpoint responsivo (900px) foi implementado via variante arbitrária do Tailwind (`min-[900px]:`) em vez do padrão de listener de `resize` em JavaScript usado no mockup — evita flash de layout incorreto na hidratação e é o padrão idiomático em Next.js/Tailwind; o comportamento visual final é idêntico ao verificado em `design/10 Árvore Genealógica.dc.html`.
4. `app/dashboard/page.tsx` envolvido com `AppShell`, recebendo o nome do tenant já carregado nessa página; conteúdo permanece placeholder, conforme escopo desta Task.
5. Refatoradas as telas de Login (`app/login/page.tsx`) e Onboarding (`app/onboarding/OnboardingWizard.tsx`), construídas na Task 1.3, para consumir `Button` e `TextField` em vez de estilos inline duplicados — elimina duplicação de lógica de estilo entre telas, conforme reaproveitamento de lógica existente.
6. Validado visualmente via preview do navegador usando uma rota temporária (`app/visualtestdashboard`, removida antes do commit) que renderiza todos os componentes dentro do `AppShell`: comparado pixel a pixel com `design/Design System.dc.html` (botões, badges, alertas, cards) e testado o breakpoint responsivo exatamente nos limites 899px (mobile) e 900px (desktop) — resultado idêntico ao design em ambos os casos.
