# Task 1.2 - Modelo multi-tenant e autenticação

## Summary
Modelo de Tenant criado no Prisma, Supabase Auth integrado (cadastro, login, logout, renovação de sessão), e middleware de isolamento multi-tenant implementado como extensão do Prisma Client, com testes unitários cobrindo acesso à própria tenant e bloqueio de acesso a tenant diferente.

## Details
1. Adicionado modelo `Tenant` ao `prisma/schema.prisma` (`ownerId` único referenciando `auth.users.id` do Supabase, `name`, `focus`, `logoUrl`). Migração `add_tenant` gerada e aplicada com sucesso no Supabase.
2. Prisma 7 removeu o middleware `$use` e não expõe mais `Prisma.dmmf` publicamente no client gerado pelo novo gerador `prisma-client`. Isso invalidou o design inicial (detecção automática de modelos tenant-scoped via DMMF). Adaptado para uma lista explícita e documentada em `lib/tenant/scoped-models.ts` (`TENANT_SCOPED_MODELS`), que tabelas de domínio futuras (aves, ninhadas) devem registrar ao adicionar `tenantId`.
3. Isolamento implementado em três módulos:
   - `lib/tenant/context.ts`: contexto de tenant por requisição via `AsyncLocalStorage`.
   - `lib/tenant/scope.ts`: função pura `scopeQueryArgs` que injeta/valida `tenantId` em `where`/`data` para todas as operações do Prisma (find*, create*, update*, upsert, delete*, count, aggregate, groupBy); lança `MissingTenantContextError` (sem contexto) ou `TenantMismatchError` (tenant divergente) — fail-closed por padrão.
   - `lib/tenant/prisma-extension.ts`: extensão `$extends` que aplica `scopeQueryArgs` em `$allModels.$allOperations` apenas para modelos na lista `TENANT_SCOPED_MODELS`.
   - `lib/prisma.ts`: cliente Prisma singleton da aplicação, já com a extensão aplicada e usando `@prisma/adapter-pg` (Prisma 7 exige adapter de driver explícito — `PrismaClient` sem `adapter` falha em tempo de compilação).
4. Integração Supabase Auth via `@supabase/ssr`: `lib/supabase/client.ts` (browser), `lib/supabase/server.ts` (Server Components/Actions), `lib/supabase/middleware.ts` + `proxy.ts` (renovação de sessão a cada requisição). `lib/auth/actions.ts` com Server Actions `signUp`, `signIn`, `signOut` (mensagens de erro traduzidas para pt-BR); `lib/auth/session.ts` com `getCurrentUser()`.
5. Arquivo gerado pelo Next.js (`middleware.ts`) renomeado para `proxy.ts` — Next.js 16 depreciou a convenção `middleware.ts` em favor de `proxy.ts` (confirmado nos docs em `node_modules/next/dist/docs`).
6. `eslint.config.mjs` ajustado para ignorar `design/**` (mockups estáticos, não código da aplicação) e `app/generated/**` (código gerado pelo Prisma).
7. Validado manualmente contra o Supabase real: cadastro de usuário de teste via `@supabase/supabase-js` funcionou (script temporário, removido após o teste).

## Important Findings
1. **Prisma 7 quebra padrões antigos**: não há mais `$use` middleware nem `Prisma.dmmf` público no client gerado pelo gerador `prisma-client`; e `PrismaClient` requer um `adapter` de driver explícito (`@prisma/adapter-pg` foi adicionado). Qualquer trabalho futuro com Prisma neste projeto deve considerar essas diferenças em vez de assumir o comportamento de versões anteriores.
2. **Confirmação de e-mail habilitada por padrão no Supabase Auth**: o cadastro de teste retornou sessão `null` até confirmação de e-mail. Isso afeta diretamente a Task 1.3 (fluxo de login/cadastro) — o usuário pode não conseguir logar imediatamente após o cadastro. Não alterei a configuração do projeto Supabase (ação externa, fora do escopo autônomo); a Task 1.3 precisa decidir como tratar isso (aviso na UI, ou pedir ao usuário para desabilitar a confirmação em ambiente de desenvolvimento).
3. **Convenção `TENANT_SCOPED_MODELS`**: toda tabela de domínio futura deve ser adicionada manualmente a essa lista além de ganhar o campo `tenantId` no schema — não é automático. Risco de esquecimento; vale considerar um teste de CI que valide que todo modelo com `tenantId` está na lista (fora do escopo desta Task).
