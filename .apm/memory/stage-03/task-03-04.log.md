# Task 3.4 - Tela Detalhe da Ninhada e fluxo "gerar filhotes"

## Summary
Tela de detalhe da ninhada implementada com preenchimento progressivo, alerta de consanguinidade (integrando o coeficiente de parentesco da Task 3.2 pela primeira vez em uma tela), e o atalho "gerar filhotes" pré-preenchendo o Novo Cadastro de Ave.

## Details
1. `lib/ninhadas/service.ts`: `getNinhada` estendida para calcular `coeficienteParentesco` do casal (via `calcularCoeficienteParentescoEntreAves`, Task 3.2) como campo derivado, seguindo o mesmo padrão de `taxaEclosao`. Nova função `listFilhotesGerados(anilhaMachoId, anilhaFemeaId)` busca aves já cadastradas com esse casal como pai/mãe.
2. `app/ninhadas/[id]/page.tsx` (guarda de autenticação/onboarding, `getNinhada` + leitura da preferência `alertasConsanguinidadeAtivados` em paralelo, depois `listFilhotesGerados`) e `FichaNinhada.tsx` (client component).
3. Status calculado da ninhada (Em curso / Risco genético / Encerrada) implementado conforme a definição operacional da Task 3.1/Integration Approach: "Encerrada" quando `filhotesNascidos` não é nulo; "Risco genético" quando a preferência do tenant está ativada E o coeficiente de parentesco é maior que zero; "Em curso" caso contrário. Reaproveita o `Badge` já existente com as variantes `risk`/`warning`/`neutral` (que já correspondem exatamente às cores do mockup para essas três situações).
4. Alerta de consanguinidade reaproveita o componente `Alert` (variante `warning`) já existente, exibido apenas quando a preferência do tenant permite e há parentesco detectado.
5. Preenchimento progressivo (`ovosPrevistos`, `ovosBotados`, `ovosFerteis`, `filhotesNascidos`): alterna entre visualização (3 cards + taxa de eclosão, quando disponível) e um formulário de edição que faz `PATCH /api/ninhadas/[id]` diretamente (sem Server Action, já que não há upload de arquivo aqui) e atualiza o estado local com a resposta — que já vem com `taxaEclosao` recalculada pelo próprio serviço.
6. Ação "Gerar filhotes na árvore": link para `/plantel/novo?paiId=...&maeId=...&especieId=...`. `NovoCadastroForm`/`app/plantel/novo/page.tsx` (Task 2.3) foram estendidos com uma prop opcional `preselecao` para ler esses query params e pré-selecionar espécie/pai/mãe, convertendo os `<select>` de pai/mãe de não controlados para controlados (mantendo editáveis pelo usuário, não travados).
7. Testado o build de produção e a suíte completa; **descoberto que o timeout padrão do Vitest (5s) começou a estourar esporadicamente** ao rodar a suíte inteira (11 arquivos, majoritariamente integração contra o Supabase remoto, executando em paralelo) — não é uma falha de lógica (os mesmos testes passam isoladamente), é contenção de rede/conexão sob carga concorrente. Aumentado para 20s em `vitest.config.ts`.
8. Validado visualmente via preview do navegador (rota temporária com dados mockados, removida antes do commit): tela de detalhe e formulário de edição conferem com `design/09 Detalhe da Ninhada.dc.html`, incluindo o alerta de consanguinidade com o coeficiente calculado.

## Compatibility Concerns
O mockup `design/09 Detalhe da Ninhada.dc.html` mostra uma grade visual de ovos individuais (ícones "fértil"/"infértil"/"eclodido" por ovo). Omitida — nosso schema (Task 3.1) armazena apenas contagens agregadas (`ovosBotados`, `ovosFerteis`, `filhotesNascidos`), não o estado de cada ovo individualmente, e a Task não pediu essa granularidade de dado. Os 3 cards agregados (Ovos/Férteis/Nascidos) cobrem a mesma informação de forma consolidada. O texto do alerta de consanguinidade também foi simplificado: o mockup descreve o tipo exato de parentesco (ex: "Rufus é pai de Amália — parentesco direto"), mas o coeficiente por si só não distingue de forma unívoca o tipo de relação (ex: 12,5% pode ser meio-irmãos OU avô-neto) — usei um texto genérico ("compartilham ancestrais próximos") em vez de inventar uma classificação de relação não pedida pela Task 3.2.
