# Task 3.3 - Telas Nova Ninhada e Lista de Ninhadas

## Summary
Formulário de Nova Ninhada implementado com alerta de consanguinidade em tempo real (não bloqueante) e seleção de casal restrita pela Trava 1; Lista de Ninhadas construída reaproveitando a classificação de status já usada na Ficha da Ninhada.

## Details
1. Novo endpoint `GET /api/parentesco?machoId=&femeaId=`: calcula o coeficiente de parentesco (Task 3.2) e lê a preferência `alertasConsanguinidadeAtivados` do tenant em paralelo, retornando ambos para o cliente decidir a exibição.
2. `lib/aves/useParentesCandidatos.ts` estendido com um parâmetro opcional `status` (retrocompatível — `undefined` preserva o comportamento das Tasks 2.3/2.4, que não filtram por status na seleção de pai/mãe genealógico). Usado com `status="ATIVO"` no formulário de Nova Ninhada para aplicar a Trava 1 na origem das opções.
3. `app/ninhadas/novo/NovaNinhadaForm.tsx`: ao selecionar macho e fêmea, busca o coeficiente via `/api/parentesco` e exibe `Alert` (variante `warning`, reaproveitado) com o percentual quando há risco e a preferência está ativa, ou `Alert` (variante `success`) confirmando ausência de parentesco direto — seguindo exatamente o padrão `showRisk`/`showSafe` do `design/08 Nova Ninhada.dc.html`. O alerta é puramente informativo: o botão "Confirmar ninhada" nunca é bloqueado por ele. Submissão via `fetch` direto a `POST /api/ninhadas` (sem Server Action, sem upload de arquivo nesta tela) — toda a validação de negócio (Trava 1, unicidade de `codNinhada`) permanece apenas no backend já existente (Task 3.1), sem duplicação no cliente.
4. `lib/ninhadas/status.ts`: `determinarStatusNinhada` extraída da Task 3.4 (estava definida localmente em `FichaNinhada.tsx`) para um módulo compartilhado, reaproveitado agora também por `app/ninhadas/page.tsx` (Lista de Ninhadas) — elimina a duplicação que existiria entre as duas telas.
5. `lib/ninhadas/service.ts`: `listNinhadas` passou a calcular `coeficienteParentesco` por ninhada (antes só `getNinhada`/detalhe fazia isso), necessário para a Lista classificar corretamente "Risco genético" — aceitável em termos de custo já que o volume de ninhadas por criatório é pequeno.
6. Lista de Ninhadas (`app/ninhadas/page.tsx`, server component): barra de progresso com semântica diferente por status — para ninhadas em andamento, `ovosFerteis/ovosBotados`; para encerradas, sempre 100% com cor distinta (`#7C9364`), replicando exatamente o comportamento observado no mockup `design/07 Lista de Ninhadas.dc.html` (a ninhada encerrada tem barra cheia mesmo com fertilidade menor que 100%).
7. Validado visualmente via preview do navegador (rotas temporárias com dados/fetch mockados, removidas antes do commit): formulário de Nova Ninhada com o alerta de risco disparando corretamente ao selecionar Rufus × Amália (25%, coincide com o par usado como exemplo no próprio mockup); Lista de Ninhadas conferida pixel a pixel com as 3 variações de status do design (risco/curso/encerrada).
