# Task 2.4 - Tela Ficha da Ave

## Summary
Tela de detalhe/edição da ave implementada, com visualização completa, edição de campos e mudança de status, reaproveitando `getAve`/`updateAve` (Task 2.1) e os componentes/badges já existentes.

## Details
1. `lib/aves/service.ts`: `getAve` estendida com `include: { especie: true, paiAve: true, maeAve: true }` para exibir nome de espécie e navegar diretamente para as fichas de pai/mãe.
2. `lib/aves/actions.ts`: refatorado para um formato compartilhado entre `createAveAction` e a nova `updateAveAction` — `requireTenantOrRedirect`, `uploadFotoSeEnviada` e `mapAveServiceError` extraídos para eliminar duplicação entre as duas Server Actions. `updateAveAction` lê o `id` de um campo oculto do formulário, delega toda a validação para `updateAve` (Task 2.1) e faz `revalidatePath` antes de redirecionar de volta à Ficha.
3. `app/plantel/[id]/page.tsx` (guarda de autenticação/onboarding, busca a ave via `getAve` dentro de `runWithTenant` — o isolamento por tenant garante que uma ave de outro criador nunca é encontrada, retornando 404 automaticamente) e `FichaAve.tsx` (client component com alternância entre modo visualização e edição).
4. Modo visualização fiel a `design/05 Ficha da Ave.dc.html`: foto/gradiente, nome, espécie · sexo, badge de status (`BIRD_STATUS_BADGE`), anilha, cards de nascimento/mutação/origem, cards de pai/mãe (linkando para a Ficha do genitor quando cadastrado no plantel), botão para a árvore genealógica (rota futura) e ações de rodapé.
5. Modo edição: mesmos campos do Novo Cadastro (Task 2.3) pré-preenchidos, mais a mudança de status via chips reaproveitando os rótulos de `STATUS_AVE_LABELS`. Os dropdowns de pai/mãe usam o hook compartilhado `useParentesCandidatos` (extraído nesta Task de `NovoCadastroForm`, que foi refatorado para consumi-lo também — elimina a duplicação de lógica de busca que existiria entre as duas telas).
6. Novos testes de integração em `lib/aves/service-update.test.ts`: mudança de status persistida e recarregada corretamente, atualização de um campo sem afetar os demais, e bloqueio ao tentar atualizar uma ave inexistente.
7. Validado visualmente via preview do navegador (rota temporária com dados e fetch mockados, removida antes do commit): modo visualização e modo edição conferem com o design; toggle de sexo reflete corretamente o valor pré-existente da ave.

## Compatibility Concerns
Duas omissões conscientes do mockup `design/05 Ficha da Ave.dc.html`, ambas por ausência de modelo de dados de suporte (mesmo padrão já usado nas Tasks 2.1/2.2):
1. **Seção "Histórico" omitida**: o mockup mostra uma linha do tempo de eventos (cadastro, vínculo a ninhadas, etc.), mas não existe nenhum modelo de eventos/log no schema atual, e a Task não pediu a criação de um. Implementar um histórico fictício ou vazio seria enganoso; melhor omitir até haver uma fonte de dados real (provavelmente decorrente do modelo de Ninhada, Stage futura).
2. **Indicador "Em ninhada" nos badges**: conforme instrução explícita da Task, não implementado — `BIRD_STATUS_BADGE`/`STATUS_AVE_LABELS` continuam representando apenas os 5 status armazenados (Ativo/Reservado/Vendido/Óbito/Fugiu).

Os links "Ver árvore genealógica completa" (`/arvore`) e "Gerar pedigree" (`/plantel/[id]/pedigree`) apontam para rotas ainda não construídas, seguindo o mesmo padrão já estabelecido na Stage 1 para os itens de navegação do `AppShell` (ex: Ninhadas, Configurações).
