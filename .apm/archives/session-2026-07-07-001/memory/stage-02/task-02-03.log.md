# Task 2.3 - Tela Novo Cadastro de Ave

## Summary
Formulário completo de cadastro de ave implementado, consumindo `createAve` via Server Action, com upload de foto para o Supabase Storage e dropdowns de pai/mãe filtrados dinamicamente por espécie e sexo.

## Details
1. Confirmado com o usuário (pausa explícita, credencial de infraestrutura): criação de um bucket novo `fotos-aves` no Supabase Storage, público, com política de INSERT para `authenticated` — evitando repetir o ponto cego do bucket `logos` (Onboarding, Stage 1) mencionado explicitamente nas instruções da Task.
2. `lib/aves/actions.ts` (`createAveAction`, Server Action): resolve usuário/tenant autenticado, faz upload opcional da foto (`fotos-aves/{tenantId}/{timestamp}.{ext}`), monta o input a partir do `FormData` e delega toda a validação de negócio para `createAve` (Task 2.1) — nenhuma regra de compatibilidade duplicada no servidor desta Task, apenas tradução de erros (`ZodError`, `AnilhaDuplicadaError`, `ParentescoInvalidoError`, `RegistroNaoEncontradoError`) para mensagens amigáveis.
3. `app/plantel/novo/page.tsx` (guarda de autenticação/onboarding + catálogo de espécies) e `NovoCadastroForm.tsx` (client component): todos os campos do modelo `Ave`, com os dropdowns de pai/mãe recarregados via `GET /api/aves?especieId=...&sexo=...` (reaproveitando o endpoint e os filtros da Task 2.2, sem nova lógica de filtragem) toda vez que a espécie selecionada muda.
4. Upload de foto com preview local (mesmo padrão do logo do Onboarding, Stage 1: `<input type="file">` oculto atrás de um `<label>` circular).
5. Toggle de sexo estendido para 3 opções (Macho/Fêmea/Não sexado) — o mockup só mostra 2, mas o schema (Task 2.1) exige suportar `NAO_SEXADO`; documentado como divergência de design abaixo.
6. Validado visualmente via preview do navegador (rota temporária com `window.fetch` mockado simulando o catálogo de espécies e o plantel existente, removida antes do commit): layout confere com `design/06 Novo Cadastro de Ave.dc.html`; confirmado via inspeção do DOM que os `<select>` de pai/mãe são populados corretamente ao trocar de espécie (o valor selecionado por padrão, "Nenhum/desconhecido", inicialmente mascarava essa checagem numa captura de tela — corrigido lendo o `outerHTML` do elemento).
7. Não fiz uma submissão real do formulário contra o Supabase de produção/dev nesta verificação (evitando uma escrita real sem necessidade, já que a lógica de criação em si já está coberta pelos testes de integração da Task 2.1). A validação da submissão real fica registrada como pendente — ver Issues.

## Issues
Não validei uma submissão real do formulário (criação de ave de fato) contra o banco, incluindo o upload real de uma foto para o bucket `fotos-aves` — a criação de teste ficou restrita aos testes de integração automatizados da Task 2.1 (que não exercitam o upload de foto, já que isso só existe nesta Task). Recomendo uma verificação manual pontual do upload de foto assim que possível, para confirmar que a política de Storage configurada pelo usuário realmente permite o INSERT.

## Compatibility Concerns
1. **Toggle de sexo com 3 opções**: `design/06 Novo Cadastro de Ave.dc.html` mostra apenas Macho/Fêmea. Estendido para incluir "Não sexado" porque o campo `sexo` do modelo `Ave` (Task 2.1) exige suportar esse terceiro valor — sem essa opção, seria impossível cadastrar uma ave não sexada pela UI. Reaproveita o mesmo estilo de toggle (`bg-tab-track`), sem introduzir um padrão visual novo.
2. **Rótulos de pai/mãe genéricos**: o mockup usa `{{ especieAtual }}` interpolado no rótulo (ex: "Pai (machos de Canário Belga no plantel)"); optei por um texto fixo mais genérico ("Pai (machos da espécie selecionada no plantel)") para simplificar a implementação sem prejuízo de clareza — pequena liberdade de redação, sem alterar layout/comportamento.
