# Task 2.2 - Tela Lista do Plantel

## Summary
`listAves` estendida com busca por nome/anilha e filtros por espécie, sexo e status, expostos via query params em `GET /api/aves`; tela de listagem construída consumindo esses endpoints, fiel ao design.

## Details
1. `lib/aves/service.ts`: `listAves` passou a aceitar `ListAvesFiltros` (`busca`, `especieId`, `sexo`, `status`), construindo um `where` do Prisma com `OR` para busca case-insensitive em `nomeApelido`/`anilha`, mais `include: { especie: true }` para exibir o nome da espécie sem consultas extras.
2. `app/api/aves/route.ts` (`GET`): passou a ler `busca`, `especieId`, `sexo`, `status` de `searchParams`, validando os enums contra `SEXO_AVE_VALUES`/`STATUS_AVE_VALUES` (`lib/aves/schema.ts`) antes de repassar ao serviço.
3. Novo endpoint `GET /api/especies` (catálogo compartilhado de espécies, não tenant-scoped) para alimentar os chips de filtro e, futuramente, os formulários de cadastro/edição.
4. Novo `lib/aves/labels.ts`: mapeamento dos enums Prisma (`StatusAve`, `SexoAve`, `OrigemAve`) para rótulos em português, reaproveitado pela UI para exibir texto e resolver a variante de badge via `BIRD_STATUS_BADGE` (Stage 1).
5. Tela `app/plantel/page.tsx` (guarda de autenticação/onboarding + busca do tenant e do catálogo de espécies) e `app/plantel/PlantelList.tsx` (client component: busca com debounce de 300ms, chips de filtro por sexo/espécie — single-select, seguindo o design — e uma segunda linha de chips por status, adicionada além do que o mockup mostra).
6. Testes de integração em `lib/aves/service-list.test.ts`: confirma isolamento por tenant na listagem (ave de outro tenant nunca aparece), busca por nome e por anilha, filtro por espécie, por status, por sexo, e combinação de múltiplos filtros.
7. Validado visualmente via preview do navegador (rota temporária com fetch mockado, removida antes do commit): layout desktop e mobile conferem com `design/04 Lista do Plantel.dc.html` (cartões, badges, chips, navegação).

## Compatibility Concerns
Duas divergências conscientes do mockup `design/04 Lista do Plantel.dc.html`, ambas justificadas pelo escopo real de dados desta Stage:
1. **Chip "Em ninhada" omitido**: o mockup mostra "Em ninhada" como um chip de filtro e como badge de status em algumas aves de exemplo, mas "Em ninhada" não é um valor armazenado em `status` (é um indicador calculado futuro, dependente do modelo de Ninhada, que a Task 2.1 explicitamente instruiu a não implementar ainda). Omitido para não expor um filtro que nunca retornaria resultados.
2. **Linha extra de chips de status** (Ativo/Reservado/Vendido/Óbito/Fugiu) foi adicionada, não presente no mockup desta tela — necessária para cumprir o requisito explícito da Task ("filtros por espécie/status"), já que o mockup só mostra status como badge informativo por linha, não como filtro. Reaproveita os tokens/cores de badge já estabelecidos, sem introduzir um padrão visual novo.
