# Task 2.1 - Modelo de dados e API de Aves

## Summary
Modelos `Ave` (tenant-scoped) e `Especie` (catálogo compartilhado) criados no Prisma, `Ave` registrada no isolamento multi-tenant, e endpoints CRUD implementados em `app/api/aves`, com as regras de negócio testadas unitariamente e por integração contra o banco real.

## Details
1. Schema Prisma: `Especie` (lista extensível, não tenant-scoped, catálogo compartilhado — `nome` único) e `Ave` (`tenantId`, `anilha` único por tenant via `@@unique([tenantId, anilha])`, `especieId`, `mutacaoCor` opcional, `sexo`/`origem`/`status` como enums Prisma, `dataNascimento` opcional, `anilhaPaiId`/`anilhaMaeId` como auto-relações nomeadas (`AvePai`/`AveMae`), `foto` como URL opcional). Migração `add_ave_e_especie` aplicada com sucesso no Supabase.
2. `Ave` registrada em `lib/tenant/scoped-models.ts` (`TENANT_SCOPED_MODELS`), herdando o isolamento automático já existente.
3. Regras de negócio isoladas em `lib/aves/`: `schema.ts` (validação Zod dos campos obrigatórios/opcionais), `compatibility.ts` (funções puras `assertPaiCompativel`/`assertMaeCompativel` — mesma espécie e sexo compatível), `errors.ts` (`AnilhaDuplicadaError`, `RegistroNaoEncontradoError`), `service.ts` (`createAve`, `listAves`, `getAve`, `updateAve`).
4. Endpoints REST em `app/api/aves/route.ts` (GET lista, POST cria) e `app/api/aves/[id]/route.ts` (GET detalhe, PATCH edita), cada um resolvendo o tenant do usuário autenticado (`lib/aves/request-context.ts`) antes de executar a operação dentro de `runWithTenant`.
5. Zod (`zod@^4`) adicionado como dependência direta para validação de entrada — não estava instalado diretamente no projeto antes.
6. **Bug de propagação de contexto descoberto e corrigido**: chamar `prisma.<modelo>.<operação>(...)` dentro do callback de `runWithTenant` sem `await` interno faz o despacho real da consulta (e a execução do middleware `$allOperations` do isolamento) ocorrer apenas no `.then()` do chamador — que roda fora da janela síncrona do `AsyncLocalStorage.run()`, perdendo o contexto de tenant e lançando `MissingTenantContextError`. Isso foi detectado pelo teste de integração (o `afterAll` falhava) e também existia silenciosamente em `listAves`/`getAve` (que faziam `return prisma...` sem `await`) — corrigido adicionando `await` explícito nas duas funções. Validado meticulosamente antes de reportar como corrigido.
7. Testes automatizados (Vitest): `compatibility.test.ts` (regras puras de parentesco), `schema.test.ts` (validação Zod dos campos obrigatórios/opcionais), `service.test.ts` (integração contra o Supabase real: unicidade de anilha por tenant — mesma anilha permitida entre tenants diferentes, bloqueada no mesmo tenant —, campos obrigatórios ausentes, e as quatro combinações de incompatibilidade de espécie/sexo em pai/mãe, além do caminho feliz com pai e mãe compatíveis). `vitest.config.ts` ajustado com `setupFiles: ["dotenv/config"]` para carregar `DATABASE_URL` nos testes de integração.
8. Validado manualmente: build de produção (`npm run build`) sem erros; servidor de dev local confirma `401` em `GET/POST /api/aves` e `GET /api/aves/[id]` sem autenticação.

## Important Findings
1. **Bug de propagação de contexto no isolamento multi-tenant** (detalhado acima, item 6): qualquer código futuro que chame o Prisma dentro de `runWithTenant` precisa fazer `await` da chamada dentro da própria função que a executa — retornar a promise sem aguardá-la internamente quebra silenciosamente o isolamento por tenant (a consulta falha com `MissingTenantContextError` em vez de vazar dados, então o modo de falha é seguro, mas ainda assim é um bug funcional). Vale revisar `lib/prisma.ts`/`lib/tenant/prisma-extension.ts` para considerar mitigar isso na própria extensão (ex: documentação mais explícita ou um wrapper que force o await), já que a Stage seguinte (Ninhadas) vai reaproveitar o mesmo padrão.
2. A entidade `Especie` foi modelada como catálogo compartilhado (não tenant-scoped) — decisão não explicitada nas instruções da Task, mas inferida do enunciado ("lista gerenciável... modele como tabela extensível", sem menção a isolamento por tenant nem instrução para registrá-la em `TENANT_SCOPED_MODELS`). Se o produto precisar de espécies customizadas por criatório, isso exigirá uma Task futura para tornar `Especie` tenant-scoped.
