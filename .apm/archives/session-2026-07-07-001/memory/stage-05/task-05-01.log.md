# Task 5.1 - Tela de Configurações

## Summary
Tela `/configuracoes` implementada seguindo `design/12 Configurações.dc.html`, cobrindo edição de perfil (nome do responsável), edição do criatório (nome/logo/foco), escrita da preferência de alertas de consanguinidade (a leitura já existia desde a Task 3.2), toggle de notificações por e-mail como placeholder, e logout.

## Details
1. Reaproveitamento confirmado antes de implementar: a rota `/configuracoes` e o ícone já estavam declarados em `components/nav/nav-items.ts` (Stage anterior), e o shell de navegação (`components/nav/AppShell.tsx`) já existia — bastou envolver a nova página nele, seguindo o mesmo padrão de `app/dashboard/page.tsx`.
2. `lib/tenant/logo.ts` (novo): extraí o upload de logo (bucket `logos`) de `lib/onboarding/actions.ts` para uma função compartilhada `uploadLogo(supabase, userId, logo)`, reaproveitada tanto pelo onboarding quanto pela nova ação de atualização do criatório — evita duplicar essa lógica de negócio, conforme a regra de reaproveitamento do projeto.
3. `lib/tenant/preferences.ts`: adicionada `atualizarAlertasConsanguinidade(tenantId, ativado)` — a leitura já existia (Task 3.2) e já é consumida por `app/api/parentesco/route.ts`, `app/ninhadas/page.tsx`, `app/ninhadas/[id]/page.tsx` e `app/arvore/[id]/page.tsx`; esta Task adiciona a primeira escrita, sem alterar o schema (campo já existente no `Tenant`).
4. `lib/settings/actions.ts` (novo): três Server Actions —`updateProfileAction` (atualiza `user_metadata.full_name` via `supabase.auth.updateUser`), `updateCriatorioAction` (atualiza nome/foco/logo do `Tenant` via `prisma.tenant.update`, reaproveitando `uploadLogo`), e `updateAlertasConsanguinidadeAction` (grava a preferência). Por isolamento multi-tenant, `updateAlertasConsanguinidadeAction` nunca recebe um `tenantId` do cliente — resolve o tenant sempre a partir da sessão autenticada (`getCurrentUser` → `prisma.tenant.findUnique({ where: { ownerId } })`), prevenindo que um id manipulado no cliente afete o tenant de outra conta.
5. `components/ui/Toggle.tsx` (novo): switch on/off reaproveitando os tokens de cor já existentes (`oliva-600`/`border`), já que não havia um componente desse tipo no design system implementado.
6. `app/configuracoes/page.tsx` + `ConfiguracoesView.tsx`: Server Component busca usuário/tenant/preferência (mesmo padrão de autenticação/onboarding das demais páginas) e repassa para o Client Component, que reproduz o layout do design — cartão de perfil com alternância para edição inline, cartão de Criatório com alternância de edição (nome/foco/logo), toggles de Preferências, e botão "Sair da conta" reaproveitando `signOut` (Task 1.2).
7. **Decisão de escopo**: a seção "Assinatura" presente no mockup foi omitida — não está nas Instruções Detalhadas nem no Output Esperado da Task, e reproduzi-la exigiria fabricar dados de cobrança (plano, preço, data de cobrança) sem nenhum sistema de assinatura real por trás, o que seria inventar uma integração não solicitada (mesmo princípio já aplicado na Task 4.4 para "Imagem"/"Compartilhar"). Documentado como Compatibility Concern.
8. O toggle de "Notificações por e-mail" foi implementado como estado local do componente (sem persistência), pois a Task tornou a persistência opcional e não há campo no schema para isso — persistir exigiria uma migração para um placeholder sem lógica de envio, o que não pareceu justificado.
9. Testes automatizados: `lib/tenant/preferences.test.ts` (integração: grava `false`/`true` no tenant real e confirma que a leitura já usada pelas telas de Ninhadas/Árvore/endpoint de parentesco reflete a mudança), `lib/tenant/logo.test.ts` (unitário, com um `SupabaseClient` mockado: bucket correto, caminho prefixado pelo id do usuário, retorno `null` em falha de upload). Não foram escritos testes de integração diretos para as Server Actions de perfil/criatório (`updateProfileAction`/`updateCriatorioAction`): elas dependem de `next/headers` `cookies()`, que não roda fora de um request Next.js real — mesma limitação já documentada na Task 4.5 para `signUpAction`. A lógica de negócio extraída para fora desse contexto (`uploadLogo`, `atualizarAlertasConsanguinidade`) está coberta.
