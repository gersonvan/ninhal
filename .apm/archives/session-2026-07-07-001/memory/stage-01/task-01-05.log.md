# Task 1.5 - Primeiro deploy público

## Summary
Aplicação publicada com sucesso na Vercel em produção, conectada ao mesmo projeto Supabase usado nas Tasks anteriores. URL pública: **https://plantelboard.vercel.app**. Fluxo de login/cadastro validado em produção; onboarding/dashboard permanecem bloqueados pela confirmação de e-mail obrigatória do Supabase Auth, conforme já esperado e antecipado nas instruções desta Task.

## Details
1. Confirmado com o usuário: conta Vercel já autenticada via CLI (`vercel whoami` → `gersonvan-1341`); projeto Supabase de produção é o mesmo já usado (`fzeyhruwmqwrcoekbwui`), sem necessidade de credenciais novas.
2. `vercel link --yes` criou o projeto `plantel_board` na Vercel. A tentativa automática de conectar o repositório GitHub (`gersonvan/ninhal`) falhou (sem acesso/repo não encontrado) — não bloqueante, o deploy foi feito diretamente via CLI a partir do diretório local, sem depender da integração GitHub.
3. Variáveis de ambiente de produção configuradas via `vercel env add` (ambiente `production`): `DATABASE_URL` (connection string do pooler do Supabase, já validada nas Tasks anteriores), `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`.
4. Adicionado `"postinstall": "prisma generate"` ao `package.json` — necessário porque o client do Prisma gerado (`app/generated/prisma`) não é versionado no git; sem isso o build da Vercel falharia por não encontrar o client.
5. Build de produção validado localmente (`npm run build`) antes do deploy, sem erros.
6. No primeiro `vercel deploy --prod`, o log de build emitiu o aviso "Detected .env file, it is strongly recommended to use Vercel's env handling instead" — indicando que o `.env` local (com credenciais reais) foi enviado junto ao pacote de deploy, além das variáveis já configuradas na Vercel. Corrigido criando um `.vercelignore` explícito (`.env`, `.env.local`, `.env.*.local`) e removendo o `.env.local` gerado automaticamente pelo `vercel link`. Um segundo deploy confirmou que o aviso desapareceu — o build passou a depender exclusivamente das variáveis de ambiente configuradas na Vercel.
7. Validação manual em produção via navegador (Chrome, ferramenta de automação): `/login` carrega e confere visualmente com o design; cadastro de um usuário de teste (autorizado explicitamente pelo usuário nesta conversa, dado que é uma escrita real no Supabase de produção) foi processado com sucesso, mas a sessão não foi estabelecida devido à confirmação de e-mail obrigatória do Supabase Auth — o usuário foi redirecionado de `/onboarding` de volta para `/login`, exatamente o comportamento já identificado nas Tasks 1.2/1.3 e antecipado nas instruções desta Task ("documente isso... não é um defeito desta Task"). Guardas de rota (`/dashboard`, `/onboarding`) confirmadas redirecionando corretamente para `/login` sem sessão.

## Validation
Build local (`npm run build`) e build de produção na Vercel sem erros. `/login` acessível publicamente e visualmente idêntico ao design (confirmado via `curl` e via navegador). Fluxo de cadastro processa corretamente em produção; onboarding/dashboard bloqueados apenas pela confirmação de e-mail obrigatória (configuração do Supabase, não defeito de deploy). Guardas de autenticação de `/dashboard` e `/onboarding` confirmadas em produção.

## Important Findings
1. O deploy inicial expôs o `.env` local (com credenciais reais) no pacote enviado à Vercel, apesar de as variáveis de produção já estarem configuradas separadamente — risco de exposição de credenciais no artefato de build. Corrigido com `.vercelignore`; recomendo que qualquer novo ambiente de deploy (preview, staging) valide a ausência desse aviso antes de considerar o deploy seguro.
2. A confirmação de e-mail obrigatória do Supabase Auth (identificada desde a Task 1.2) agora bloqueia o fluxo completo também em produção, não apenas em desenvolvimento. Antes de qualquer teste de aceitação com usuários reais ou da Stage de testes E2E, é necessário decidir: desabilitar a confirmação de e-mail no painel do Supabase Auth, ou implementar uma tela de "confirme seu e-mail" na aplicação.
