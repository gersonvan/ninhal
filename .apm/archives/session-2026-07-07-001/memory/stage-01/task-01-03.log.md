# Task 1.3 - Telas de Login e Onboarding do criatório

## Summary
Telas de Login/Cadastro e o fluxo de Onboarding (3 passos) implementadas seguindo o design aprovado, com upload de logo para o Supabase Storage e criação do registro de Tenant ao final, redirecionando para um Dashboard placeholder.

## Details
1. Configurados os tokens de design (`app/globals.css`) e fontes (`app/layout.tsx`, via `next/font/google`: Manrope, Source Serif 4, Space Grotesk) a partir de `design/Design System.dc.html` — cores, tipografia e espaçamentos usados como referência direta.
2. `app/login/page.tsx`: componente cliente com abas Entrar/Criar conta e o estado de recuperação de senha ("Esqueci minha senha"), fiel ao layout de `design/01 Login.dc.html` (fundo escuro com gradiente radial, cartão claro, mesma hierarquia de campos e textos).
3. Ajustadas as Server Actions de autenticação (`lib/auth/actions.ts`) do formato `(email, password)` para o padrão `(prevState, formData)` compatível com `useActionState`/`<form action>`; adicionada `requestPasswordResetAction` (não fazia parte da Task 1.2, necessária para a tela de recuperação do design).
4. `app/onboarding/OnboardingWizard.tsx`: wizard de 3 passos (nome do criatório, foco de criação com seleção múltipla de espécies, upload de logo com preview), fiel a `design/02 Onboarding.dc.html`, incluindo os indicadores de progresso e o pré-selecionado "Canários" no passo 2.
5. `lib/onboarding/actions.ts` (`completeOnboardingAction`): valida o usuário autenticado, opcionalmente envia o logo ao bucket público `logos` do Supabase Storage (confirmado previamente criado pelo usuário), e faz `upsert` do `Tenant` (criado na Task 1.2) com `ownerId`, `name`, `focus` e `logoUrl`; redireciona para `/dashboard`.
6. `app/dashboard/page.tsx` e `app/page.tsx`: guarda de autenticação (redireciona para `/login` sem sessão) e de onboarding (redireciona para `/onboarding` sem `Tenant`); dashboard é uma tela placeholder mínima com botão de sair, conforme escopo explícito da Task ("uma tela placeholder é suficiente").
7. Validado o fluxo visualmente via preview do navegador: tela de Login/Cadastro (screenshot conferido pixel a pixel com o design), e os 3 passos do Onboarding (usando um bypass temporário de autenticação controlado por env var, removido antes do commit final — usado apenas para verificação visual, já que o cadastro real esbarrou no rate limit de e-mail do Supabase, ver Issues).
8. `eslint.config.mjs` de Task 1.2 permaneceu válido, sem ajustes necessários.

## Validation
`npx tsc --noEmit`, `npx eslint .` e `npx vitest run` (14/14) sem erros após as mudanças. Tela de Login validada visualmente contra o design (screenshot). Onboarding (3 passos) validado visualmente passo a passo via preview do navegador. Cadastro real de usuário não pôde ser validado ponta a ponta nesta Task devido a rate limit do Supabase Auth (ver Issues) — o mecanismo de cadastro em si já foi validado funcionalmente na Task 1.2.

## Issues
Ao testar o cadastro real via navegador, o Supabase retornou `429 email rate limit exceeded` (limite do plano gratuito atingido pelos testes de cadastro acumulados nas Tasks 1.2 e 1.3). Não é um defeito no código — a criação de usuário já foi comprovada funcional na Task 1.2. Recomendo aguardar a janela de rate limit expirar (ou configurar um provedor de SMTP customizado no Supabase) antes de testes manuais adicionais de cadastro.

## Compatibility Concerns
Removi o campo "Registro / CNPJ (opcional)" do passo 1 do Onboarding, presente no mockup (`design/02 Onboarding.dc.html`), porque o modelo de `Tenant` (Task 1.2) e o objetivo/output explícitos desta Task listam apenas nome do criatório, foco de criação e logo como dados a coletar — não há campo correspondente no schema para persistir um registro/CNPJ, e adicionar um campo de formulário sem persistência real criaria uma UX enganosa. Sinalizo essa divergência do design para decisão do Manager: se o CNPJ for necessário, requer uma Task específica para adicionar o campo ao schema do Tenant.

## Important Findings
1. A confirmação de e-mail obrigatória no Supabase Auth (identificada como achado importante na Task 1.2) segue sem tratamento na UI — um usuário recém-cadastrado não consegue logar imediatamente. Isso deve ser resolvido antes de testes de aceitação com usuários reais (ajuste de configuração no painel do Supabase, ou tela de aviso "confirme seu e-mail").
2. O bucket `logos` do Supabase Storage foi criado pelo usuário durante esta Task, mas as políticas de acesso (Storage RLS policies) para permitir upload por usuários autenticados não foram verificadas nesta Task — se o upload falhar em uso real, é provável que falte uma policy de INSERT para `authenticated` no bucket.
